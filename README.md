# CarHero Backend

NestJS backend for the CarHero platform. The codebase is organized by feature modules and follows a pragmatic clean architecture layout:

```txt
src/
  app.module.ts
  main.ts
  config/                 Application-level configuration
  core/                   Shared guards, decorators, filters, pipes, enums, utils
  database/               Seeders and database bootstrap helpers
  modules/
    <feature>/
      <feature>.module.ts
      domain/             Entities, repository contracts, domain services
      application/        DTOs, use cases, application services
      infrastructure/     Mongoose schemas, repositories, external adapters
      presentation/       HTTP controllers and gateways
```

## Main Modules

| Module | Responsibility |
|---|---|
| `auth` | User/provider auth, OTP, refresh tokens, pending registrations, logout records |
| `admin` | Admin auth, platform settings, stats, admin accounts, canonical admin APIs |
| `users` | Customer profile and admin user management |
| `providers` | Provider profile, onboarding, approval, runtime status, provider search |
| `services` | Service catalog, categories, emergency services, admin service management |
| `orders` | Instant orders and scheduled bookings backed by the same `orders` collection |
| `status-history` | Status timeline for orders and scheduled bookings |
| `audit` | Immutable admin activity log for sensitive actions |
| `wallet` | Wallets, transactions, payouts, earnings, loyalty points |
| `subscriptions` | Subscription plans and user subscriptions |
| `vehicles` | User vehicles, maintenance records, reminders |
| `reviews` | Ratings, comments, provider responses |
| `chat` | Conversations, messages, uploads |
| `notifications` | In-app/push notification records |
| `whatsapp` | WhatsApp Web integration and QR view |

## Current API Notes

Admin user management is centralized under:

```http
/admin/users
```

Admin subscription management should use the canonical routes:

```http
GET    /admin/subscription-plans
POST   /admin/subscription-plans
PATCH  /admin/subscription-plans/:id
DELETE /admin/subscription-plans/:id
GET    /admin/subscriptions
```

Orders and scheduled bookings share the same `orders` collection:

```txt
Instant order       -> isScheduled=false
Scheduled booking   -> isScheduled=true + scheduledAt
```

Order status transitions are centralized in:

```txt
src/modules/orders/domain/services/order-state-machine.ts
```

Frontend can query allowed status actions with:

```http
GET /orders/:id/status-transitions
```
